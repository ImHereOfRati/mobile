# ImHere Mobile OMO Harness

이 하네스는 ImHere Mobile 저장소에서 작업하는 에이전트가 반드시 따라야 할 기준이다.

## 1. 작업 태도

1. 추측하지 않는다.
2. 해석이 여러 개면 조용히 하나를 고르지 않는다.
3. 더 단순한 해결책이 있으면 먼저 그쪽을 택한다.
4. 사용자가 요청한 범위만 수정한다.
5. 모든 변경 줄은 사용자 요청과 직접 연결되어야 한다.

## 2. 변경 원칙

1. 기본값은 최소 변경이다.
2. 기능 추가, 리팩터링, 이름 변경, 포맷 정리는 가능하면 분리한다.
3. 인접 코드까지 임의로 정리하지 않는다.
4. 내 변경으로 인해 생긴 미사용 import, 변수, 함수만 정리한다.
5. 설명할 수 없는 추상화, 단일 사용처를 위한 범용화는 만들지 않는다.

## 3. 구조 원칙

1. 화면 코드는 화면 관심사만 가진다.
2. 비즈니스 규칙과 외부 연동은 UI 밖에 둔다.
3. UI, 애플리케이션 흐름, 도메인 규칙, 저장소/외부 시스템 접근의 경계를 유지한다.
4. 기능 단위 구조를 우선하고, 공통화는 반복이 확인된 뒤에만 한다.
5. 숨겨진 전역 상태와 암묵적 사이드이펙트를 피한다.

## 4. Flutter/Dart 규칙

1. 이름은 역할이 드러나야 한다.
2. `null` 허용은 정말 필요한 경우에만 사용한다.
3. 화면에서 비즈니스 로직을 직접 실행하지 않는다.
4. 로딩, 성공, 실패 상태를 분명히 표현한다.
5. 권한 거부, 빈 상태, 오류 상태를 정상 흐름만큼 중요하게 다룬다.

## 5. 오류 처리 규칙

1. 의미 없는 문자열이나 원시 예외로 실패를 표현하지 않는다.
2. 호출 측이 처리 가능한 오류 형태를 유지한다.
3. 사용자 표시용 오류와 내부 진단용 오류를 구분한다.
4. 실제 가능한 실패 경로를 숨기지 않는다.
5. 화면 곳곳에 중복 `try-catch`를 남발하지 않는다.

## 6. 테스트 및 검증 규칙

1. 버그 수정은 가능하면 재현 테스트를 먼저 만든다.
2. 새로운 규칙은 가능한 한 테스트로 고정한다.
3. 테스트 계층은 단위 테스트, 기능 흐름 테스트, 통합 검증 중 가장 비용 대비 가치가 큰 것을 고른다.
4. 테스트가 없더라도 분석, 빌드, 핵심 흐름 점검 중 하나는 수행한다.
5. 검증하지 못한 것은 검증하지 못했다고 분명히 말한다.

## 7. 보안 및 개인정보 규칙

1. 위치, 연락처, 토큰, 전화번호는 최소 범위로만 다룬다.
2. 로그, 테스트 데이터, 문서에 민감정보 원문을 남기지 않는다.
3. 권한 요청은 편의보다 명시성을 우선한다.
4. 비밀값을 임의로 생성하거나 파일에 기록하지 않는다.

## 8. 문서 및 인코딩 규칙

1. 새 규칙이나 반복 실수는 채팅에만 남기지 말고 저장소 문서에 반영한다.
2. 문서는 현재형으로 작성한다.
3. 저장소의 문서와 소스 파일은 UTF-8을 유지한다.
4. 한글이 포함된 텍스트는 인코딩 깨짐 가능성을 항상 의식한다.

## 9. 리뷰 기준

1. 안정성
2. 가독성
3. 유지보수성
4. 테스트 가능성
5. 개발 속도
