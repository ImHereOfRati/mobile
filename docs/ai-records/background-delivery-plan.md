# 도착 자동 알림 백그라운드 실행 전략

## 배경
- 사용 시나리오:
  1. 앱을 닫는다
  2. 자가용/버스/지하철/기차/배 등으로 30분 이상 이동한다
  3. 특정 위치를 지난다
  4. 서버에 알람 발송을 요청한다
- 현재 문제:
  - 위치 감지 이후 백그라운드 작업이 자동 종료되어 서버 요청이 중간에 실패할 수 있음

## 핵심 판단
- `항상 허용`만으로는 충분하지 않다
- `항상 허용`은 백그라운드 위치 감지의 전제 조건
- 하지만 서버 요청 안정성은 별도의 문제이며, `배터리 최적화 제외 + 백그라운드 작업 구조 개선 + 재시도 전략`이 필요하다

## 권한별 역할
- 항상 허용: 앱이 닫힌 상태에서도 도착 위치를 감지하기 위한 필수 권한
- 배터리 최적화 제외: Doze / App Standby / OEM 절전 정책으로 인한 후속 작업 종료 완화
- 알림 권한: 상태 표시, 실패/보류 안내, 필요 시 Foreground Service 노출 기반
- 부팅 완료 수신: 재부팅 후 활성 도착 알림 재등록
- wake lock: 이벤트 직후 CPU를 잠깐 깨워 후속 처리 기회 확보

## 기술 판단
- 문제의 본질은 "위치 감지"보다 "감지 후 서버 요청 완료 보장"에 더 가깝다
- 따라서 다음 3단계로 분리해야 한다:
  1. 도착 감지
  2. 도착 이벤트 영속 저장
  3. 서버 요청 재시도 처리

## 권장 아키텍처
- geofence 이벤트 수신 시 바로 모든 작업을 처리하지 않는다
- 백그라운드 콜백에서는 최소한만 수행:
  - 이벤트 ID 저장
  - 중복 방지 상태 저장
  - 처리 큐 적재
- 실제 서버 요청은 재시도 가능한 백그라운드 실행 단위로 위임:
  - WorkManager 또는 동등한 네이티브 재시도 작업
  - 필요 시 짧은 Foreground Service 승격
- 성공 시:
  - 기록 저장
  - geofence 비활성화
  - 큐 완료 처리
- 실패 시:
  - 재시도 큐에 남김
  - 앱 재실행/네트워크 복구/재부팅 시 재처리

## 운영 원칙
- 목표를 "즉시 보내기"에서 "결국 반드시 보내기"로 재정의
- 실시간성이 중요하면 Foreground Service 전략을 추가 검토
- 제조사별 절전 정책 예외를 고려한 진단 UI 필요

## UX 설명 원칙
- 항상 허용: "앱을 닫고 이동 중이어도 도착을 감지하려면 필요해요."
- 배터리 최적화 제외: "도착을 감지한 뒤 자동 전송이 중간에 끊기지 않게 해요."

## 실행 계획
1. 백그라운드 로그 포인트 세분화
2. 종료 지점 분석
3. geofence callback 처리 최소화
4. pending event 큐 설계
5. WorkManager 기반 재시도 구조 설계
6. 권한 UX를 기능 활성화형으로 재정리
7. 설정 화면에 진단 상태 카드 추가
