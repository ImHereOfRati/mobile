# ImHere UI 전환 계획

## 목표
- 사용자가 30초 안에 첫 번째 알림을 만들 수 있게 한다
- 권한은 앱 사용 조건이 아니라 기능 활성화 수단으로 재설계한다
- `지오펜스 앱` 인상을 `위치 기반 자동 연락 서비스`로 전환한다
- 메인 화면을 설정 중심이 아니라 `내 알림 관리` 중심으로 바꾼다

## 원칙
- 현재의 전체적인 시각 인상은 유지한다
- 밝은 회색 배경 + 흰 카드 + 블루 포인트를 유지한다
- GmarketSans + BMHANNAAir 조합을 유지한다
- 기술 제약보다 사용자가 얻는 결과를 먼저 설명한다
- 권한은 필요한 순간에만 요청한다
- 내부 구현 용어와 사용자 노출 용어를 분리한다

## 개발 원칙

### OOP 지향
- 화면, 상태, 권한 판단, 카피 결정, 라우팅 판단을 역할별 객체/구성요소로 분리한다
- View 에서 권한 판단 로직을 직접 분기하지 않고 정책 객체 또는 상태 모델로 캡슐화한다
- 사용자 노출 용어와 내부 도메인 용어의 매핑 책임은 한곳에 둔다
- 공통 UI 컴포넌트, 상태 카드, 섹션 헤더, 안내 카드가 있으면 신규 구현보다 재사용을 우선한다

### TDD 지향
- 1차 마일스톤은 가능한 범위에서 TDD로 진행한다
- 구현 전에 실패하는 테스트를 먼저 작성하고, 최소 구현 후 리팩터링한다
- 우선 테스트 대상:
  - 용어 매핑
  - 권한 상태 -> 준비 상태 변환
  - 자동 전송 준비 여부 판단
  - Permission Prep 진입 조건
  - 주요 라벨/상태 문구 회귀
- UI는 골든 테스트보다 상태 기반 위젯 테스트를 우선한다
- 라우팅, 권한 정책, 상태 변환은 단위 테스트를 우선한다

### 최소 권한 원칙
- 최소한의 권한만 다룬다
- 1차에서 핵심적으로 다루는 권한은 아래로 한정한다
  - 최소 위치 권한
  - 위치 항상 허용
  - 배터리 최적화 제외
- 푸시/연락처 권한은 기존 기능 유지 범위에서만 다루고, 핵심 퍼널 재설계 대상에서는 우선순위를 낮춘다
- 권한 요청은 지금 필요한 기능 기준으로만 발생해야 한다
- 동일 기능을 더 낮은 권한으로 제공할 수 있다면 그 구조를 우선한다

### 코드 구조 기준
- private 메서드가 많아지면 책임이 과도하게 뭉쳤는지 먼저 의심한다
- private 메서드가 5개 이상이면 아래를 검토한다
  - 권한 판단 로직 분리
  - 상태 해석 객체 분리
  - 공통 UI 컴포넌트 추출
  - 카피/프레젠테이션 모델 추출
- build 메서드가 100줄을 넘어가면 공통 요소 추출을 검토한다
- 파일이 200줄을 넘어가면 역할 분리 또는 공통 요소 추출을 검토한다
- 단, 무조건 쪼개지 말고 역할이 실제로 분리되는지를 기준으로 판단한다

### View / Theme 원칙
- View 에서 color 하드코딩을 지양하고 Theme 기반 색상을 사용한다
- 색상은 `Theme.of(context).colorScheme`, `textTheme`, 공통 theme extension 을 우선 사용한다
- 신규 상태 카드/배너/버튼도 기존 theme 토큰 안에서 해결하는 것을 우선한다
- 예외적으로 새 색상이 필요하면 개별 View 내부 상수보다 공통 theme 또는 style 계층으로 올리는 것을 먼저 검토한다

## 범위 구분
- 1차: 백엔드 변경 없이 가능한 UX 전환
- 2차: 앱 내부 모델 확장으로 가능한 기능 확장
- 3차: 백엔드/API 확장이 필요한 제품 고도화
