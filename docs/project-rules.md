# OMO Rules For ImHere Mobile

이 문서는 ImHere Mobile 팀이 따를 `OMO(Oh My OpenAgent)` 기준의 협업 규칙을 정의한다.

`OMO`의 핵심은 사람과 에이전트가 같은 저장소에서 함께 일하더라도, 결과물이 더 단순하고 더 안전하며 더 검증 가능해야 한다는 점이다.

## 1. OMO의 목표

- 에이전트는 속도를 높이기 위한 도구이지, 판단 책임을 대체하지 않는다.
- 모든 변경은 사람이 이어서 읽고 수정할 수 있어야 한다.
- 결과보다 과정의 추적 가능성을 중시한다.
- 자동화된 작업일수록 범위, 근거, 검증이 더 분명해야 한다.

## 2. 기본 원칙

- 작은 변경을 우선한다.
- 한 번에 한 가지 문제를 해결한다.
- 추측으로 구조를 넓히지 않는다.
- 기존 맥락을 확인한 뒤 수정한다.
- 설명할 수 없는 추상화는 추가하지 않는다.
- 더 단순한 해결책이 있으면 먼저 그쪽을 검토한다.
- 불확실한 해석이 둘 이상이면 조용히 선택하지 않는다.

## 3. 사람과 에이전트의 역할

- 사람은 방향, 우선순위, 최종 승인 책임을 가진다.
- 에이전트는 탐색, 구현, 정리, 검증을 빠르게 수행한다.
- 요구사항이 모호하면 에이전트는 임의 확장보다 짧은 질문을 우선한다.
- 요구사항이 충분히 명확하면 에이전트는 제안만 하지 말고 실제 변경까지 수행한다.
- 에이전트는 자신이 바꾼 범위를 분명히 설명해야 한다.

## 4. 작업 단위 규칙

- 하나의 작업은 하나의 목적을 가져야 한다.
- 기능 추가, 리팩터링, 이름 변경, 포맷 정리는 가능하면 분리한다.
- 큰 작업은 단계별로 나누고 각 단계마다 검증 가능해야 한다.
- 중간 산출물이 남는 작업은 문서, 로그, 테스트 중 하나로 근거를 남긴다.
- 성공 조건이 모호하면 구현 전에 검증 기준부터 정리한다.

## 5. 코드 작성 규칙

- 기본값은 최소 변경이다.
- 새 파일, 새 타입, 새 추상화는 필요가 명확할 때만 만든다.
- 함수와 클래스 이름은 역할이 드러나야 한다.
- 화면 코드에는 화면 관심사만 남긴다.
- 비즈니스 규칙과 외부 연동 코드는 UI 밖에 둔다.
- 같은 의미의 개념을 여러 이름으로 부르지 않는다.
- 단일 사용처를 위한 범용화는 만들지 않는다.
- 내 변경으로 생긴 미사용 import, 변수, 함수만 정리한다.
- 인접 코드를 임의로 같이 개선하지 않는다.

## 6. 에이전트 친화적 코드 규칙

- 파일 구조는 사람이 찾기 쉬워야 하고, 에이전트도 검색만으로 흐름을 따라갈 수 있어야 한다.
- 진입점, 주요 상태 전환, 외부 연동 경계는 이름만 보고도 추적 가능해야 한다.
- 숨겨진 전역 상태, 암묵적 사이드이펙트, 과도한 매직 값 사용을 피한다.
- 복잡한 분기에는 짧게라도 의도를 남긴다.
- 코드 생성물과 수동 작성 코드는 역할이 섞이지 않게 관리한다.
- 변경된 모든 줄은 사용자 요청과 직접 연결되어야 한다.

## 7. 아키텍처 선택 규칙

- 변경이 적고 흐름이 단순한 기능은 단순한 계층 구조를 우선한다.
- 외부 연동이 많고 정책 변화가 잦은 기능은 경계를 분리한 구조를 우선한다.
- 기능 단위로 폴더를 묶고, 공통 레이어보다 도메인 맥락을 먼저 드러낸다.
- UI, 애플리케이션 흐름, 도메인 규칙, 외부 저장소 접근의 경계는 유지한다.
- 아키텍처는 프로젝트 전체 통일성보다 변경 비용 절감에 기여해야 한다.

## 8. 오류 처리 규칙

- 원시 `Exception` 또는 의미 없는 오류 메시지로 실패를 표현하지 않는다.
- 호출 측이 처리할 수 있도록 실패 의미를 분명한 타입, 코드, 상태로 전달한다.
- 화면 단에서 `try-catch`를 반복 남발하지 말고 오류 처리 흐름을 일관되게 유지한다.
- 사용자에게 보여줄 오류와 내부 진단용 오류를 구분한다.
- 불가능한 상황을 과하게 방어하지 말되, 실제로 발생 가능한 실패 경로는 숨기지 않는다.

## 9. 테스트 및 검증 규칙

- 새로운 규칙이나 버그 수정은 가능하면 테스트로 고정한다.
- 테스트는 단위 테스트, 기능 흐름 테스트, 통합 검증 중 가장 값이 큰 계층을 선택한다.
- 단순 입력 검증과 비즈니스 규칙 검증은 같은 테스트 목적에 섞지 않는다.
- 테스트 이름은 조건, 행동, 기대 결과가 드러나야 한다.
- 테스트가 없더라도 최소한 분석, 빌드, 핵심 흐름 점검 중 하나는 남긴다.
- 검증하지 못한 항목은 완료로 포장하지 않는다.

## 10. 문서화 규칙

- 문서는 코드의 대체물이 아니라 의사결정의 기록이어야 한다.
- 복잡한 기능은 구현 전에 용어, 경계, 실패 조건부터 정리한다.
- 새 규칙이 생기면 채팅 기록에만 남기지 말고 저장소 문서로 승격한다.
- 문서는 현재형으로 작성하고, 희망 사항은 TODO로 분리한다.
- 작업 중 드러난 반복 실수나 새 패턴은 관련 규칙 문서에 즉시 반영한다.

## 11. 인코딩 및 텍스트 규칙

- 저장소의 문서와 소스 파일은 UTF-8을 기본으로 유지한다.
- 한글이 포함된 문서, 로그, 테스트 데이터는 인코딩 깨짐 가능성을 항상 의식한다.
- 인코딩이 중요한 설정이나 출력 포맷은 암묵값에 기대지 말고 명시한다.

## 12. 리뷰 규칙

- 리뷰는 결과물의 모양보다 위험을 먼저 본다.
- 리뷰 기준은 안정성, 가독성, 유지보수성, 테스트 가능성 순으로 본다.
- 에이전트가 만든 코드도 사람 코드와 동일한 기준으로 리뷰한다.
- 리뷰에서 합의된 규칙은 반복 설명하지 말고 문서에 반영한다.

## 13. 보안 및 민감정보 규칙

- 토큰, 위치, 연락처, 전화번호 같은 민감정보는 최소 범위로만 다룬다.
- 로그, 테스트 데이터, 스크린샷, 문서에도 민감정보 원문을 남기지 않는다.
- 권한과 개인정보 처리는 편의보다 명시성을 우선한다.
- 에이전트는 비밀값을 새로 만들거나 임의로 파일에 기록하지 않는다.

## 14. Git 규칙

- 커밋은 의도를 설명해야 한다.
- 관련 없는 변경은 함께 묶지 않는다.
- 사람이 만들었든 에이전트가 만들었든, 머지 전에는 변경 범위를 다시 읽는다.
- 자동 생성 파일은 실제 소스 변경과 구분해서 검토할 수 있어야 한다.

## 15. 금지 사항

- 확인하지 않은 가정을 사실처럼 반영하지 않는다.
- 설명 없는 대규모 구조 변경을 하지 않는다.
- 사용되지 않는 범용화 코드를 미리 만들지 않는다.
- 실패를 숨기거나 로그만 남기고 넘어가지 않는다.
- 사람이 이해하지 못하는 상태로 작업을 끝내지 않는다.
- 콘솔 임시 출력, 의미 없는 디버그 흔적, 설명 없는 매직 값을 남기지 않는다.

## 16. 의사결정 우선순위

- 보안
- 안정성
- 단순성
- 검증 가능성
- 개발 속도

## 17. OMO 한 줄 원칙

- 에이전트가 빠르게 만들고, 사람이 끝까지 이해할 수 있어야 한다.
